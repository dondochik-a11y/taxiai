_Составлен 2026-07-17 по результатам ревизии кода (два разведочных агента: web + bot)._

Принцип приоритизации: сначала то, что доводит до пользователя **уже собранные реальные данные** (радарный кэф), потом UX-дыры, потом полировка.

## Фаза 0 — фундамент (блокирует остальное)

**0.1. Докатить незакоммиченную работу по кэфу.**
В working tree уже лежат: каскад `radar_stale` / `radar_near` в `surge_service.py`, новый `kef_profile_service.py` (эмпирический прогноз кэфа по (район, день недели, час) из `kef_observations`), правки `inference.py`, тесты `test_kef_forecast.py`, обновлённые типы web. Доделать, прогнать тесты, закоммитить, задеплоить на VPS. Без этого фичи ниже показывают synthetic-заглушки.

**0.2. Непрерывность сбора данных.**
Ночной провал 23:00–01:20 показал: сон Mac'а останавливает и эмулятор, и launchd-свипы. Лечение: `sudo pmset -a sleep 0 displaysleep 10` (или `caffeinate` в run_emulator.sh) + буфер/ретрай в `post_readings` (сейчас свип против недоступного API теряет данные).

## Фаза 3 — связка и полировка

- Deep links бот ↔ PWA (кнопка в боте открывает карту с нужным районом; в PWA — кнопка «открыть бота»).
- Единая учётка: приглашение связать аккаунт в момент, когда это полезно (после первого `/where` — «в веб-версии есть карта»).
- Уведомление о всплеске кэфа в домашнем районе по реальным радарным данным (сейчас PRESHIFT_ALERT работает от synthetic demand).
