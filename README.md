# TaxiAI

AI-копилот для водителя такси: карта спроса, прогноз, рекомендации "куда ехать", автоматический учёт финансов, AI-чат и Telegram-бот. Полное описание архитектуры и принятых решений — в `README`-секциях ниже и в комментариях кода; это MVP-каркас, работающий полностью на синтетических данных и без единого внешнего API-ключа (см. "Provider mode" ниже).

## Стек
- **apps/api** — FastAPI + PostgreSQL/PostGIS + Redis + scikit-learn
- **apps/web** — Next.js 16 (App Router) + TypeScript + Tailwind, PWA
- **apps/bot** — Telegram-бот (aiogram), тонкий клиент над backend
- **infra** — docker-compose для локальной разработки

## Быстрый старт

Требуется Docker + Docker Compose.

```bash
cp .env.example .env          # все внешние ключи опциональны — по умолчанию всё работает в mock-режиме
make up                        # поднимает postgres+postgis, redis, api, worker, bot
make migrate                   # применяет схему БД (создаёт все таблицы + сидит районы Москвы)
make seed                      # генерирует 90 дней синтетической истории (спрос/погода/пробки/рейсы/поездки)
make train                     # обучает модель прогноза спроса на синтетических данных
```

Backend будет доступен на `http://localhost:8000` (`/docs` — Swagger UI). Воркер (`app/jobs/scheduler.py`) сам подхватит обученную модель и раз в 15 минут будет пересчитывать прогнозы; раз в 5 минут — добавлять новую "живую" синтетическую точку данных (имитация реального сбора).

Frontend запускается отдельно (быстрее для итерации, чем через compose):

```bash
cd apps/web
npm install
npm run dev   # http://localhost:3000
```

Первым делом откройте `/onboarding`, чтобы создать профиль водителя — без него дашборд не сможет запрашивать персональные рекомендации/финансы.

Telegram-бот: заполните `TELEGRAM_BOT_TOKEN` в `.env` (получить у @BotFather), `make up` пересоздаст контейнер `bot`.

## Provider mode — почему всё работает без ключей

Каждая внешняя интеграция (OpenAI, погода, авиарейсы, пробки, праздники, метро) спрятана за интерфейсом в `apps/api/app/providers/`. По умолчанию `PROVIDER_MODE=mock` — все данные читаются из уже сгенерированной синтетической истории или отдаются шаблонным движком (для GPT-анализа и чата). Как только появится реальный ключ — впишите его в `.env`, смените соответствующий `*_PROVIDER_MODE=live`, **и обязательно выполните `docker compose -f infra/docker-compose.yml up -d` (не `restart`!)** — `restart` не перечитывает `.env`, только `up -d` пересоздаёт контейнер с новыми переменными.

### Как получить бесплатные ключи

| Сервис | Зачем | Бесплатный лимит | Где получить |
|---|---|---|---|
| **isDayOff.ru** | реальные праздники РФ | без ограничений, ключ не нужен | ✅ уже подключено и работает |
| **TomTom Traffic** | реальные пробки, раз в 2 часа на район (чётные часы — live, нечётные — синтетика: 130 районов × 24 часа не влезают в лимит) | 2 500 запросов/день | ✅ уже подключено и работает |
| **AviationStack** | реальное расписание рейсов SVO/VKO/DME, раз в сутки | 100 запросов/мес | ✅ уже подключено и работает |
| **OpenSky Network** | реальное обнаружение "самолёт заходит на посадку", раз в 15 мин | OAuth2 client credentials, без жёсткого лимита для зарегистрированных | ✅ уже подключено, ждём реального захода борта в узкую зону (~15км) вокруг аэропорта, чтобы увидеть первую запись |
| **OpenWeatherMap** | реальная погода | 60 запросов/мин, 1 млн/мес | ⏳ ключ вписан, ждём активации (обычно до ~2 часов после регистрации) |
| **Яндекс.Такси (виджет-API)** | реальный кэф: цены поездок по районам, кэф = цена / базовая цена района | лимиты по договорённости (`PRICING_POLL_MINUTES` настраивает частоту) | ⏳ код готов (`/v1/surge/current`), нужен clid+apikey — заявка на https://yandex.ru/dev/taxi/ |

После получения ключа: впиши его в `.env` (`OPENWEATHER_API_KEY=...` и т.д.), поставь соответствующий `*_PROVIDER_MODE=live`, затем **обязательно** `docker compose -f infra/docker-compose.yml up -d api worker` (не `restart` — он не перечитывает `.env`).

AviationStack и OpenSky — два разных источника рейсов, специально не взаимозаменяемые: AviationStack даёт расписание (кто когда должен прилететь, раз в сутки — лимит 100/мес не позволяет чаще), OpenSky — реальные ADS-B позиции бортов в воздухе (кто заходит на посадку прямо сейчас, раз в 15 минут). Оба пишут в одну и ту же таблицу `airport_flights` и одинаково участвуют в сигнале "спрос у аэропорта растёт при частых прилётах".

## Что реально проверено, а что нет

- **Backend + Docker**: полностью проверено вживую (`docker compose up`, `make migrate`, `make seed`, `make train`, все API-эндпоинты, чат, рекомендации, создание поездки + AI-анализ, все 4 подключённых внешних источника данных) — в процессе было найдено и исправлено 7 реальных багов, которые не ловились до реального запуска: permission на entrypoint.sh, пустые env-переменные, путь к seed-файлу внутри контейнера, рассинхронизация часовых меток погоды/трафика с ML-пайплайном, порядок проверки намерений в чат-боте, несовпадение статусов рейсов AviationStack с нашей схемой enum, ISO-строки дат вместо datetime-объектов при записи рейсов в БД.
- **Реальные данные**: isDayOff (праздники), TomTom (пробки), AviationStack (расписание рейсов) — подключены и проверены вживую на реальном Postgres. OpenSky (реальные позиции бортов) — подключён, OAuth2 и API проверены вживую, но за время проверки в узкой зоне вокруг аэропортов не оказалось садящегося борта (это ожидаемо и не баг — просто нет данных прямо сейчас). OpenWeatherMap — ключ подключён, код готов, но ключ ещё не активировался у провайдера.
- **Frontend**: проверен вживую в headless-браузере (онбординг → дашборд с картой → рекомендация «Куда ехать?» → чат → поездки → финансы), найденный hydration-баг (чтение localStorage в первом рендере) исправлен через `useSyncExternalStore` (`apps/web/lib/useStoredUserId.ts`). Если геолокация недоступна (например, телефон по LAN-http — geolocation требует secure context), рекомендация считается от домашнего района из профиля.
- **Районы**: справочник покрывает все 125 официальных районов Москвы (включая Зеленоград) + хабы вокзалов/аэропортов, всего 130 строк (`packages/shared/constants/moscow_districts.json`, миграция `0004` досеивает недостающие в существующую БД).
- **Тесты**: `apps/api/tests/` пуст — автотестов нет, только ручная проверка.

## Дальше (не входит в этот MVP-проход)
- Метро (закрытия/аварии) и календарь мероприятий (концерты/футбол) — свободного API нет, остаются синтетическими.
- Профиль/настройки водителя после онбординга, мультигород, полноценный offline-service-worker для PWA.
- Автоматическое переобучение модели (`app/jobs/retrain_model.py` — стаб, не запускается по расписанию).
- Автотесты (pytest для backend, что-то вроде Playwright для frontend).
